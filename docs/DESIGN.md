 # 随行档 (Suixingdang) 设计文档
 
 > 一个长在你自己服务器上、靠浏览器访问、用对话驱动的私人文件中枢。
 > 核心价值：公司端零安装、零痕迹，家里端常驻同步，中间用 agent 让你用自然语言管理文件。
 
 ---
 
 ## 1. 问题与定位
 
 ### 1.1 核心痛点
 
 外包员工的跨场景文件使用困境：
 
 - 在家学习/工作产生的文件，需要带到公司使用
 - 公司的文件和资料，需要带回家使用
 - 作为外包人员，工作不稳定，**不想在公司电脑安装任何网盘客户端**，离职清理麻烦
 
 ### 1.2 核心矛盾
 
 传统网盘（百度网盘、坚果云、Syncthing 等）解决的是"同步"问题，却制造了最想避免的东西——**痕迹**：客户端安装、配置文件、同步缓存、后台进程，离职时都得清理。
 
 ### 1.3 产品定位
 
 **随行档**：一个长在你自己服务器上、靠浏览器访问、用对话驱动的私人文件中枢。
 
 ### 1.4 三条差异化
 
 | 差异点 | 说明 |
 |--------|------|
 | 零痕迹 | 公司电脑只开浏览器，不装任何东西。离职时在服务器一键吊销访问权 + 清浏览器记录即可 |
 | 即问即得 | 不用翻文件夹，用自然语言告诉 agent 意图，它找到文件、传好、通知你 |
 | 懂你的文件 | agent 索引过文件名和内容，能分类、能提醒、能建议 |

 ## 2. 系统架构
 
 ### 2.1 整体拓扑
 
 ```
 [家里电脑]                        [你的服务器]                     [公司电脑]
   可以装软件                        域名 + IP                        零安装
   常驻守护进程                      存储中枢                         纯浏览器
   本地 agent                       Web 服务                         登录访问
       │                            AI 大脑 (索引/RAG)                  │
       │         自动双向同步                │                        │
       └──────────────────────────▶ 存储中枢 ◀──────────────────────────┘
                                    Web 服务            HTTPS
                                    AI 大脑
                                    权限/审计
 ```
 
 ### 2.2 三端角色（不对称设计）
 
 这种不对称直接对应你的约束条件，是整个产品的地基。
 
 **家里电脑（你的地盘）**
 
 - 可以装软件，你完全拥有
 - 轻量守护进程：文件夹监听 + 本地 agent
 - 后台自动运行，双向同步
 
 **服务器（唯一真正的服务）**
 
 - 存储中枢：所有文件的中央仓库
 - Web 服务：提供浏览器访问界面
 - AI 大脑：文件索引、语义搜索、内容问答
 - 权限与审计：访问控制、操作日志
 
 **公司电脑（零安装）**
 
 - 只开浏览器
 - 登录域名 → 找文件 / 对话 / 上传下载
 - 没有客户端、没有配置文件、没有同步缓存
 
 ### 2.3 离职清理链路
 
 这是最初痛点的闭环设计：
 
 ```
 服务器后台点"吊销公司访问"
         │
         ▼
 公司那台机器上不存在任何需要删的东西
         │
         ▼
 只剩浏览器历史记录
         │
         ▼
 清一下浏览器记录即可，干干净净走人
 ```
 
 agent 还能当"离职助手"，提醒你别漏该清理的东西。

 ## 3. Agent 智能层设计
 
 ### 3.1 Agent 定位
 
 把 agent 当成一个有手有眼、能操作文件系统的助手。它不只是聊天，它能实际执行操作。
 
 ### 3.2 Agent 工具集
 
 | 工具名 | 功能 | 说明 |
 |--------|------|------|
 | `search_files` | 语义 + 关键字找文件 | "上个月那份报价"也能命中 |
 | `sync` | 发起同步/推送 | 按自然语言意图执行文件搬运 |
 | `auto_tag` | 自动归类 | 学习/工作/项目/个人，自动去重 |
 | `summarize` | 文档摘要 | 不打开就能了解内容 |
 | `qa` | 内容问答 | RAG 检索，"这份合同的关键条款" |
 | `guard` | 敏感文件提醒 | 同步前扫描，提醒而非硬拦 |
 | `cleanup_hint` | 清理建议 | 识别长期不用、该归档、该删的文件 |
 
 ### 3.3 Guard 层详解
 
 Guard 是文件的"安检岗"，核心职责：**在文件被搬运之前，先替你看一眼这个东西该不该动、往这个方向动合不合适。**
 
 #### 3.3.1 设计原则
 
 - **提醒而非硬拦**：根据确认的敏感度（不高），guard 做智能提醒即可，用户确认后放行
 - **方向感知**：同样的文件，往公司带和往家带，判断标准不同
 - **不打扰**：只在对劲度低于阈值时才提醒，正常文件静默放行
 
 #### 3.3.2 典型场景
 
 **防止误带公司机密回家**
 
 > 你在家让 agent "把工作文件夹全同步过来"，里面混了一份带客户名单的内部表格。
 > guard 扫到"客户""合同金额""内部机密"这类信号，拦一下问："这份看起来是公司敏感文件，确定要同步到你家吗？"
 
 **防止把个人隐私带去公司**
 
 > 你周一匆忙把整个文件夹推到公司，里面有简历、银行流水、体检报告。
 > guard 提醒："这几份是个人隐私文件，是不是别推过去？"
 
 **防止密钥泄露**
 
 > 代码目录里有个 `.env` 文件写着 API key、数据库密码。
 > guard 提示："这个文件包含凭据，建议加到排除名单，不同步。"
 
 #### 3.3.3 实现思路
 
 - **规则引擎（Phase 2）**：文件名模式匹配（`.env`、`id_rsa`、`*.pem`）、目录白/黑名单、扩展名过滤
 - **内容扫描（Phase 2）**：正则匹配密钥格式（AWS key、数据库连接串）、关键词匹配（身份证号、银行卡号、合同金额）
 - **LLM 判断（Phase 3）**：对边界模糊的文件，让 LLM 读一眼给出风险评级和建议

 ## 4. 技术选型
 
 ### 4.1 选型总表
 
 | 层 | 选型 | 理由 |
 |----|------|------|
 | 部署方式 | Docker Compose | 一键起，好迁移好备份 |
 | 后端框架 | FastAPI (Python) | AI 生态最顺，tool-calling 原生支持 |
 | 文件存储 | 本地磁盘起步 | 文档代码量级完全够，量大再上 MinIO |
 | 数据库 | SQLite 起步 | 单用户场景足够，省一个组件 |
 | 向量库 | Chroma | 轻量、嵌入式，给语义搜索用 |
 | 前端 | 轻量 SPA (React) | 文件浏览 + 对话窗就够 MVP |
 | 反向代理/HTTPS | Caddy | 自动 Let's Encrypt，配置近乎为零 |
 | 家里守护进程 | Python watchdog | 文件夹监听 + 自动同步 |
 | LLM | OpenAI API (按量付费) | 最快跑通智能层 |
 
 ### 4.2 选型决策依据
 
 **为什么 FastAPI 而非 Node/Go？**
 
 Agent 的核心是调用 LLM 和做 tool-calling，Python 生态（LangChain、OpenAI SDK、文件处理库）最成熟，迭代速度最快。单用户场景下性能不是瓶颈。
 
 **为什么 SQLite 而非 Postgres？**
 
 单用户、单服务器场景，SQLite 的零运维优势远大于并发能力。数据量上来或需要多服务并发写入时，可无缝迁移到 Postgres（ORM 层隔离）。
 
 **为什么 Chroma 而非 Qdrant/Milvus？**
 
 Chroma 是嵌入式向量库，不需要额外起一个服务进程，Docker Compose 少一个容器。单用户文件量（数千到数万）完全在其舒适区。
 
 **为什么 Caddy 而非 Nginx？**
 
 Caddy 的自动 HTTPS 是杀手级特性——配置文件里写上你的域名，它自动签发和续期 Let's Encrypt 证书，零手动操作。Nginx 需要手动配 certbot，多一层维护。
 
 **为什么本地磁盘而非 MinIO/S3？**
 
 文档 + 代码 + 少量媒体，总量大概率在几十到几百 GB。本地磁盘直接挂载最简单。MinIO 的价值在于 S3 兼容接口和横向扩展，当前阶段用不上，但架构预留了升级路径。

 ## 6. 数据模型
 
 ### 6.1 核心表（SQLite）
 
 **files（文件表）**
 
 | 字段 | 类型 | 说明 |
 |------|------|------|
 | id | TEXT (UUID) | 主键 |
 | path | TEXT | 相对路径 |
 | name | TEXT | 文件名 |
 | size | INTEGER | 文件大小（字节）|
 | content_hash | TEXT | 内容哈希（去重用）|
 | mime_type | TEXT | MIME 类型 |
 | tag | TEXT | 自动分类标签 |
 | uploaded_at | DATETIME | 上传时间 |
 | modified_at | DATETIME | 修改时间 |
 | source | TEXT | 来源（home/company/manual）|
 | guard_status | TEXT | guard 检测状态（safe/warning/blocked）|
 
 **sync_events（同步事件表）**
 
 | 字段 | 类型 | 说明 |
 |------|------|------|
 | id | TEXT (UUID) | 主键 |
 | file_id | TEXT | 关联文件 |
 | direction | TEXT | 方向（home_to_server / server_to_home / upload）|
 | status | TEXT | 状态（pending/completed/failed）|
 | created_at | DATETIME | 创建时间 |
 
 **chat_history（对话历史表）**
 
 | 字段 | 类型 | 说明 |
 |------|------|------|
 | id | TEXT (UUID) | 主键 |
 | role | TEXT | user/assistant |
 | content | TEXT | 消息内容 |
 | tool_calls | TEXT | 工具调用记录 (JSON) |
 | created_at | DATETIME | 时间 |
 
 **access_tokens（访问令牌表）**
 
 | 字段 | 类型 | 说明 |
 |------|------|------|
 | id | TEXT (UUID) | 主键 |
 | label | TEXT | 标签（如 "公司电脑"）|
 | token_hash | TEXT | 令牌哈希 |
 | expires_at | DATETIME | 过期时间 |
 | revoked | BOOLEAN | 是否已吊销 |
 | last_used_at | DATETIME | 最后使用时间 |

 ## 7. API 设计
 
 ### 7.1 认证

 ### 7.2 文件操作
 
 | 方法 | 路径 | 说明 |

 ## 8. 安全设计
 
 ### 8.1 传输安全
 
 - 全站 HTTPS（Caddy 自动签发 Let's Encrypt 证书）
 - API 通信全部走 HTTPS
 
 ### 8.2 认证与授权
 
 - JWT 令牌认证，短有效期 + 刷新机制
 - 公司端访问建议开启 TOTP 双因子
 - 访问令牌可按设备签发、可单独吊销（离职清理用）
 
 ### 8.3 Guard 防护
 
 - 敏感文件同步前提醒
 - 凭据文件（.env、密钥）默认排除
 - 内容敏感度扫描（Phase 2）
 
 ### 8.4 审计
 
 - 所有文件操作记录日志
 - 所有登录记录日志（IP、时间、设备）
 - 日志可在 Web 界面查看

 ## 9. 分阶段实施路线
 
 ### Phase 0：地基（先能用）
 
 **目标**：服务器跑起来，浏览器能传文件、看文件、下文件。
 
 - [ ] Docker Compose 编排
 - [ ] FastAPI 后端骨架
 - [ ] SQLite 数据库初始化
 - [ ] JWT 认证
 - [ ] 文件上传/下载/列表 API
 - [ ] 前端文件浏览页（列表 + 上传按钮）
 - [ ] Caddy 反向代理 + HTTPS
 
 **交付标准**：打开域名，登录后能上传、浏览、下载文件。到这一步其实已经能用了——纯网页传文件、零安装。
 
 ### Phase 1：同步 + 基础对话
 
 **目标**：家里自动同步，agent 能听懂同步指令。
 
 - [ ] 家里守护进程（watchdog 文件夹监听）
 - [ ] 自动双向同步逻辑
 - [ ] Agent 工具集：`search_files`、`sync`
 - [ ] 前端对话窗口
 - [ ] 对话历史存储
 
 **交付标准**：在家改了文件，自动同步到服务器；在对话里说"把 XX 同步过来"，agent 执行。
 
 ### Phase 2：索引与智能
 
 **目标**：文件全量索引，语义搜索，自动归类，Guard 上线。
 
 - [ ] Chroma 向量库集成
 - [ ] 文件全量索引（文件名 + 内容）
 - [ ] 语义搜索
 - [ ] 自动分类标签（auto_tag）
 - [ ] Guard 规则引擎 + 内容扫描
 - [ ] 自动去重（content_hash）
 
 **交付标准**：搜"上个月那份报价"能命中；上传时自动归类；敏感文件同步前提醒。
 
 ### Phase 3：贴心层
 
 **目标**：从工具变成助手。
 
 - [ ] 文档摘要（summarize）
 - [ ] 内容问答（qa，RAG）
 - [ ] LLM 驱动的 Guard 判断
 - [ ] 智能同步建议（基于使用习惯）
 - [ ] 清理建议（cleanup_hint）
 - [ ] 离职清理助手
 
 **交付标准**：不用打开文件就能问"这份合同的关键条款"；agent 主动建议"你周一开会要用，要不要现在推到公司"。

 ## 10. 运维与备份
 
 ### 10.1 备份策略
 
 - 文件存储目录定期打包备份（cron + rsync 到第二位置）
 - SQLite 数据库定期 dump
 - Docker Compose 配置 + .env 纳入版本控制（密钥用环境变量）
 
 ### 10.2 监控
 
 - 容器健康检查
 - 磁盘空间告警
 - 同步失败通知
 
 ### 10.3 升级路径
 
 | 组件 | 当前 | 升级触发条件 | 升级目标 |
 |------|------|-------------|---------|
 | 数据库 | SQLite | 需要多服务并发写入 | PostgreSQL |
 | 存储 | 本地磁盘 | 容量接近上限 | MinIO (S3 兼容) |
 | 向量库 | Chroma | 索引量超十万级 | Qdrant |
 | LLM | OpenAI API | 想降成本/保隐私 | Ollama + Qwen (本地) |
